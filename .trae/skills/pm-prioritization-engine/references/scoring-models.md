## RICE

**全称**：Reach × Impact × Confidence / Effort

**适用场景**：团队有一定数据基础，需要跨需求横向比较，并向管理层解释排期依据。

### 各维度评分细则

#### 触达（Reach）

定义：在给定时间周期内，该功能会影响多少用户。

**有数据时**：直接使用实际用户数或百分比
```
触达 = 预计受影响的用户数（如某功能影响 20% 的 MAU = 10万用户中的 2万）
```

**无数据时（经验估算档位）**：
| 档位 | 触达值 | 说明 |
|------|-------|------|
| 全量 | 1000 | 100% 用户都会看到/使用 |
| 高 | 500 | >50% 用户 |
| 中 | 200 | 10-50% 用户 |
| 低 | 50 | <10% 用户 |
| 极低 | 10 | 仅特定场景（如管理员功能）|

**注意**：Reach 是绝对数，不是百分比，这样才能横向比较。

#### 影响（Impact）

定义：对单个用户的影响深度，对应核心指标的改善幅度。

| 分值 | 描述 |
|------|------|
| 3 | 大幅提升核心指标（转化率、留存率等提升 ≥10%）|
| 2 | 明显改善用户体验（能在用研中听到强烈正反馈）|
| 1 | 轻微改善（会有正面影响，但不显著）|
| 0.5 | 很难判断是否有影响 |

**校准技巧**：先确定 Impact=2 的基准需求（团队公认有价值的），再比较其他需求是更高还是更低。

#### 信心（Confidence）

定义：对触达和影响估算的把握程度。

| 信心值 | 依据 |
|-------|------|
| 100% | 有 A/B 测试数据或类似功能的历史数据 |
| 80% | 有定性用户研究（访谈/问卷）支撑 |
| 50% | 纯经验判断，无数据 |
| 20% | 完全假设 |

**用途**：信心系数会大幅拉低"感觉很好但没有依据"的需求，鼓励用数据说话。

#### 工作量（Effort）

定义：实现该需求需要的总人周数（所有角色加总）。

**估算规范**：
- 包含：设计 + 前端 + 后端 + 测试 + 上线
- 单位：人周（1人×1周=1人周）
- 粒度：允许 0.5 人周的精度

**无法估算时**：用 T-shirt 尺码并换算
| T-shirt | 人周 |
|---------|------|
| XS | 0.5 |
| S | 1 |
| M | 3 |
| L | 8 |
| XL | 20 |

### RICE 分值解读

```
RICE 分 = (R × I × C%) / E

举例：
- R=500（高触达）× I=2（明显影响）× C=80%（有研究支撑）/ E=2（2人周）
- RICE = 500 × 2 × 0.8 / 2 = 400
```

**相对比较**：RICE 分本身没有绝对意义，只用于同一需求池内的横向比较。

## ICE

**全称**：Impact × Confidence × Ease

**适用场景**：快速排序、早期产品、直觉驱动的团队，需要在 30 分钟内出结果。

### 各维度评分细则（均为 1-10 分）

#### 影响（Impact）

该需求实现后，对业务目标的贡献程度：
- 10：能直接驱动核心 KPI 大幅增长
- 7-9：明显正向影响核心指标
- 4-6：有帮助，但不是主要驱动力
- 1-3：边际影响，聊胜于无

#### 信心（Confidence）

对影响评估的确信程度：
- 10：有强力数据支撑
- 7-9：有用户研究/类比数据
- 4-6：团队直觉，有一定经验支撑
- 1-3：纯假设

#### 易实现度（Ease）

实现的难易程度（工作量的反向）：
- 10：1-2天可完成
- 7-9：1周内
- 4-6：1-2周
- 1-3：超过1个月

### ICE 使用建议

1. **团队一起打分**：ICE 主观性强，集体讨论比个人打分更可靠
2. **先独立，后讨论**：每人先单独打分，再汇总讨论分歧项
3. **不要追求精确**：ICE 的价值是"足够快地区分出明显高低"，不是精确排名
4. **前 3 名可信，后面可疑**：ICE 排名越靠后越不稳定，建议只相信前 1/3

## 成本收益分析

**适用场景**：有明确财务目标，或需要向业务方/投资人解释资源投入的合理性。

### 基本框架

```
ROI = (预期收益 - 实现成本) / 实现成本 × 100%

或更简单的优先级分：
优先级分 = 预期收益 / 实现成本
```

### 收益量化方法

**直接收益**：
- 新增付费用户 × ARPU
- 减少流失 × LTV
- 提升转化率 × 当前漏斗数据

**间接收益（难以量化时，用等级代替）**：

| 等级 | 含义 |
|------|------|
| 5 | 战略性价值（支撑核心竞争力）|
| 4 | 重要业务指标直接受益 |
| 3 | 间接带动关键指标 |
| 2 | 体验改善，长期保留效果 |
| 1 | 锦上添花 |

### 成本量化

- 开发成本 = 人周数 × 团队平均周成本
- 机会成本 = 做了这个不能做其他的代价（选填）
- 维护成本 = 上线后预计每季度的维护工时（选填）

## 模型组合使用建议

### 两模型交叉验证

最常用的组合是 RICE + ICE，用于相互验证：

```
一致（均高）→ 高优先级，直接排 Now
一致（均低）→ 低优先级，进入 Later
RICE高/ICE低 → 重要但不紧急，排 Next
RICE低/ICE高 → 快赢项，适合填补版本空档
```

### 三模型综合

当有 Kano 数据时，加入第三维度：

| RICE | ICE | Kano | 建议 |
|------|-----|------|------|
| 高 | 高 | Must-have | 立即做，绝对 Now |
| 高 | 高 | Delighter | Now，打造差异化 |
| 高 | 低 | Performance | 慎重：成本高但用户关注，拆小做 |
| 低 | 高 | Must-have | 快做：修复核心问题的低成本手段 |
| 任意 | 任意 | Indifferent | 砍掉或推迟到明年 |
| 任意 | 任意 | Reverse | 立即从路线图移除 |

## 常见陷阱

### 1. 工作量被系统性低估

**现象**：打分时工作量总偏小，导致 RICE/ICE 分虚高，版本永远做不完。

**解法**：
- 使用历史数据校准（过去3个版本实际工作量 vs 估算工作量的比值）
- 默认在估算上乘以 1.5 的"悲观系数"
- 将测试和上线工作量显式计入

### 2. 影响分膨胀

**现象**：团队对自己的需求都觉得"很重要"，Impact 普遍偏高，失去区分度。

**解法**：
- 引入锚定需求（强制把某个公认普通的需求定为 Impact=1，相对比较）
- 要求每个 Impact≥3 的需求提供数据或研究支撑

### 3. 信心分被忽视

**现象**：打完 RICE 的 R×I 后，C 一律填 80%，失去意义。

**解法**：
- C 只有两种来源：有数据（≥80%）或没数据（50%）
- 明确要求：C=80% 及以上的需求，必须能说出数据来源

### 4. Kano 模型被直觉替代

**现象**：没有用户数据，靠 PM 直觉填 Kano 分类，导致所有功能都是"期望功能"。

**解法**：没有用户数据时，Kano 仅供参考，不作为强决策依据。建议同时设计一个快速调研（参考 survey-designer）。

### 5. 路线图排满没有缓冲

**现象**：Now 版本工作量刚好等于可用资源，一旦出现问题就延期。

**解法**：Now 版本容量设为可用资源的 70-80%，留 20-30% 作为应急缓冲。
