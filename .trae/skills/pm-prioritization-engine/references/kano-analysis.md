## Kano 五类功能解析

| 类别 | 中文 | 特征 | 策略 |
|------|------|------|------|
| Must-be | 必要功能 | 没有=极度不满；有了=理所当然，不加分 | 必须做，但不是差异化来源 |
| Performance | 期望功能 | 做得越好越满意，线性关系 | 持续优化，这是竞争主战场 |
| Attractive | 兴奋功能 | 有了超出预期；没有不在意 | 这是差异化的机会 |
| Indifferent | 无差异功能 | 有没有都无所谓 | 砍掉，把资源用在其他类别 |
| Reverse | 反向功能 | 有了反而不满意 | 立即取消，别再做了 |

### 随时间的衰减规律

功能类别会随时间变化：
- 今天的兴奋功能（Attractive）→ 明天的期望功能（Performance）→ 后天的必要功能（Must-be）
- 典型案例：WiFi（早期是兴奋功能）→ 现在是必要功能
- 启示：差异化窗口是有限的，兴奋功能要趁早做

## Kano 问卷设计

每个功能问两道题：正向题（functional）+ 反向题（dysfunctional）

### 标准题型

**正向题**（如果有这个功能，你感觉如何？）：
```
○ 我喜欢这样
○ 这是理所当然的
○ 我无所谓
○ 我勉强接受
○ 我不喜欢这样
```

**反向题**（如果没有这个功能，你感觉如何？）：
```
○ 我喜欢这样
○ 这是理所当然的
○ 我无所谓
○ 我勉强接受
○ 我不喜欢这样
```

## 结果计算方法

### 分类矩阵（Kano Evaluation Table）

| 正向↓ / 反向→ | 喜欢 | 理所当然 | 无所谓 | 勉强接受 | 不喜欢 |
|-------------|------|---------|-------|---------|-------|
| 喜欢 | 可疑 | A（兴奋）| A | A | P（期望）|
| 理所当然 | R（反向）| 可疑 | I（无差异）| I | M（必要）|
| 无所谓 | R | I | I | I | M |
| 勉强接受 | R | I | I | I | M |
| 不喜欢 | R | R | R | R | 可疑 |

### 汇总计算

对每个功能统计所有受访者的分类结果，取出现最多的类别：

```
功能 A 的分类结果（n=100人）：
- Must-be（M）：45人
- Performance（P）：30人
- Attractive（A）：15人
- Indifferent（I）：10人

→ 分类为 Must-be（M占比最高）
```

**争议处理**：当 M 和 P 接近时，综合考虑绝对数量和战略意图。

### 满意度系数

用于量化功能对满意度的影响幅度：

```
满足系数（SI）= (A + P) / (A + P + M + I)
不满足系数（DSI）= -1 × (P + M) / (A + P + M + I)
```

- SI 越接近 1：有了这个功能，满意度提升越大
- DSI 越接近 -1：没有这个功能，满意度下降越大

## 四象限战略解读

以满足系数（SI）为 X 轴，不满足系数（|DSI|）为 Y 轴，将功能分为四象限：

```
           高 |DSI|（没有会很不满）
               │
    [必须做好]  │  [核心竞争力]
    Must-be   │  Performance
               │
───────────────┼───────────────  高 SI（有了会很满意）
               │
    [砍掉]     │  [差异化机会]
  Indifferent  │   Attractive
               │
           低 |DSI|
```

### 各象限策略

**核心竞争力区（高SI + 高|DSI|）**：Performance 功能
- 策略：持续投入，越好越有竞争优势

**差异化机会区（高SI + 低|DSI|）**：Attractive 功能
- 策略：选择性投入，可以成为品牌记忆点

**基础保障区（低SI + 高|DSI|）**：Must-be 功能
- 策略：达到及格线即可，过度投入浪费资源

**无效投入区（低SI + 低|DSI|）**：Indifferent 功能
- 策略：从路线图移除，把资源用在其他区域

## 无调研数据时的近似方法

当没有用户调研数据时，可以用以下方式近似 Kano 分类，但**必须标注为[经验估算]**：

### 快速判断问题

对每个候选功能，问自己：

1. **如果这个功能出故障/下线了，用户会投诉吗？**
   - 会大量投诉 → Must-be
   - 会有部分抱怨 → Performance
   - 可能没人注意 → Indifferent 或 Attractive

2. **竞品都有这个功能吗？**
   - 竞品都有 → 可能是 Must-be
   - 只有少数竞品有 → 可能是 Performance
   - 几乎没有竞品做 → 可能是 Attractive 或 Indifferent

3. **用户曾经主动要求过这个功能吗？**
   - 高频反复提到 → Must-be 或 Performance
   - 偶尔提到 → Performance 或 Attractive
   - 从未提到 → Indifferent 或 Attractive（惊喜型）

### 经验参照库

| 功能类型 | 通常的 Kano 分类 |
|---------|---------------|
| 安全/隐私保护 | Must-be |
| 核心业务流程 | Must-be |
| 页面加载速度 | Performance |
| 搜索精准度 | Performance |
| 个性化推荐 | Attractive（正在变 Performance）|
| 深色模式 | Attractive（部分用户群是 Must-be）|
| 节日特效/彩蛋 | Attractive |
| 隐藏的高级设置 | Indifferent（大多数用户）|
| 强制引导/弹窗 | Reverse |
| 复杂的注册流程 | Reverse |
