## 5-Why 分析法

最常用的归因方法。通过反复追问"为什么"，穿透表象找到根因。

### 使用规则

1. 从可观察的偏差事实出发
2. 每一层 Why 必须基于事实或合理推理，不能跳跃
3. 至少追 3 层，通常 5 层能触及根因
4. 如果一层有多个原因，分叉为多条链路分别追问
5. 停止标准：到了可改变的流程/机制/行为层面

### 常见陷阱

| 陷阱 | 表现 | 修正 |
|------|------|------|
| 过早停止 | Why 只追了 1-2 层就下结论 | 继续追问，直到触及机制层面 |
| 归因到人 | "因为小明没做好" | 追问"是什么机制让这件事依赖小明个人？" |
| 归因到态度 | "因为不够重视" | 追问"是什么导致优先级判断出了问题？" |
| 循环归因 | A 导致 B，B 导致 A | 找到循环的起点，从机制层面打破 |
| 跳跃归因 | 从现象直接跳到根因，跳过中间环节 | 补全中间的逻辑链条 |

### 追问模板

```
偏差事实：[具体描述偏差]
├─ Why 1：[直接原因]
│  ├─ Why 2：[再追一层]
│  │  ├─ Why 3：[继续追]
│  │  │  ├─ Why 4：[接近根因]
│  │  │  │  └─ Why 5：[根因] → 改进动作
│  │  │  └─ [分支原因] → 改进动作
│  │  └─ [分支原因] → 改进动作
│  └─ [分支原因] → 改进动作
└─ [另一条直接原因链路]
```

## 鱼骨图

适合问题原因复杂、涉及多方因素的场景。将原因按类别梳理。

### 六大类别（6M）

| 类别 | 英文 | 在产品研发中的对应 |
|------|------|-----------------|
| 人 | Man | 团队能力、经验、沟通、人员变动 |
| 机 | Machine | 工具、环境、基础设施、自动化 |
| 料 | Material | 需求质量、输入数据、第三方依赖 |
| 法 | Method | 流程、规范、评审机制、测试策略 |
| 环 | Environment | 市场变化、政策、竞品动态、天灾 |
| 测 | Measurement | 指标定义、监控覆盖、数据准确性 |

### 使用步骤

1. 把偏差事实写在鱼头
2. 画出 6 条鱼骨（六大类别）
3. 在每条鱼骨上列出该类别下的可能原因
4. 对每个原因评估影响程度（高/中/低）
5. 对高影响原因用 5-Why 深挖

### 输出格式

```markdown
### 鱼骨图归因：[偏差事实]

| 类别 | 原因 | 影响程度 | 是否需要深挖 |
|------|------|---------|-----------|
| 人 | 关键开发请假，替补不熟悉代码 | 高 | 是 → 5-Why |
| 法 | 没有代码交接 SOP | 中 | 是 |
| 料 | 需求文档缺少边界条件 | 高 | 是 → 5-Why |
| 测 | 测试用例未覆盖该场景 | 高 | 是 → 5-Why |
| 机 | 测试环境与生产环境配置不一致 | 中 | 是 |
| 环 | 第三方 API 当天升级，未提前通知 | 低 | 否（不可控） |
```

## 故障树分析

适合事故类复盘（P0/P1），从故障结果往上追因。

### 使用步骤

1. 顶层事件：描述故障现象
2. 逐层向下分解：每层列出导致上层事件的所有可能子事件
3. 标注逻辑关系：AND（所有子条件都满足才导致）/ OR（任一子条件满足即导致）
4. 找到基础事件（不可再分解的底层原因）
5. 对基础事件设计防护措施

### 输出格式

```markdown
### 故障树：[顶层故障描述]

[顶层故障]
├── OR ──┤
│        ├── [子原因 A]
│        │    ├── AND ──┤
│        │    │         ├── [基础事件 A1] → 防护措施
│        │    │         └── [基础事件 A2] → 防护措施
│        └── [子原因 B]
│             └── [基础事件 B1] → 防护措施
```

## 时间线分析法

适合过程中的延期/阻塞类问题。按时间轴还原发生了什么。

### 使用步骤

1. 列出关键时间节点（计划 vs 实际）
2. 在每个节点标注：发生了什么、谁做了什么决定、信息是否及时同步
3. 标记偏离计划的转折点
4. 对每个转折点做 Why 分析

### 输出格式

```markdown
### 时间线分析

| 日期 | 计划事项 | 实际发生 | 偏差 | 关键决策/事件 |
|------|---------|---------|------|-------------|
| 3/1 | 需求评审 | 需求评审（正常）| 无 | — |
| 3/5 | 开发启动 | 开发启动（正常）| 无 | — |
| 3/10 | 联调开始 | 第三方接口未就绪 | ⚠️ 阻塞 | 决定先开发其他模块 |
| 3/14 | — | 第三方接口交付 | 延迟4天 | 开始联调 |
| 3/15 | 提测 | 联调中 | 延迟 | — |
| 3/18 | — | 提测 | 延迟3天 | 压缩测试时间 |
| 3/20 | 上线 | 测试发现P0 Bug | ⚠️ | 决定修复后再发 |
| 3/22 | — | 上线 | 延迟7天 | — |

→ 转折点：3/10 第三方接口延迟，导致后续链式延期
→ 根因：未提前确认第三方交付时间，缺少 SLA 机制
```

## 场景选择建议

| 场景 | 推荐方法 | 原因 |
|------|---------|------|
| 单一偏差、原因明确 | 5-Why | 快速直接，5分钟出结论 |
| 多因素交叉导致 | 鱼骨图 + 5-Why | 先梳理类别再深挖 |
| 线上事故/故障 | 故障树 | 适合分析防护缺口 |
| 项目延期 | 时间线分析 | 还原过程找转折点 |
| 复杂项目（多偏差） | 先时间线，再 5-Why | 先理清事实再归因 |
