# 多角色评审检查清单

本文件是各角色在评审 PRD / 原型时的详细检查项。评审时逐项过，未通过的项目纳入评审报告。

---

## 一、产品经理检查清单

### 需求完整性
- [ ] 背景和业务上下文是否交代清楚
- [ ] 产品目标是否明确且可量化（有指标、有目标值、有时间）
- [ ] 目标用户是否定义清晰（角色、特征、核心诉求）
- [ ] 功能范围是否有明确的"做 / 不做"边界
- [ ] 是否有版本规划（MVP → 迭代路径）

### 用户故事
- [ ] 每个功能模块是否有用户故事
- [ ] 用户故事格式是否规范（作为 XX，我希望 XX，以便 XX）
- [ ] 用户故事是否覆盖所有角色的核心场景

### 流程与逻辑
- [ ] 主流程是否完整闭环（有起点、有终点、无断裂）
- [ ] 子流程是否在需要时展开
- [ ] 状态流转是否完整（每个状态都有入口和出口）
- [ ] 有没有逻辑矛盾或自相矛盾的描述

### 优先级
- [ ] 功能清单是否有优先级标注（P0/P1/P2）
- [ ] P0 功能是否真的是核心必备
- [ ] 优先级排序的依据是否合理

---

## 二、研发工程师检查清单

### 数据与接口
- [ ] 数据字典是否完整（字段名、类型、必填、说明、示例值）
- [ ] 字段的校验规则是否定义（长度、格式、范围、唯一性）
- [ ] 外部接口依赖是否列出（接口名、方向、协议、超时策略）
- [ ] 数据同步方式是否明确（实时 / 定时 / 事件驱动）

### 技术可行性
- [ ] 有没有技术上不可实现或成本极高的需求
- [ ] 并发场景是否有说明（同时编辑、重复提交、竞态条件）
- [ ] 是否需要幂等处理
- [ ] 事务一致性要求是否明确
- [ ] 是否涉及大批量数据处理（需要考虑分页、异步）

### 性能与架构
- [ ] 性能指标是否定义（响应时间、并发量、可用性）
- [ ] 数据存储方案是否合理（选型、索引、生命周期）
- [ ] 是否需要缓存策略
- [ ] 第三方服务不可用时的降级方案

### 工作量评估
- [ ] 前端工作量评估
- [ ] 后端工作量评估
- [ ] 联调周期评估
- [ ] 外部依赖联调风险

---

## 三、测试工程师检查清单

### 验收标准
- [ ] 每个核心功能是否有验收条件
- [ ] 验收条件是否可执行（能直接转化为测试用例）
- [ ] 验收条件是否覆盖正向流程和反向流程

### 异常路径
- [ ] 网络异常是否有处理
- [ ] 服务超时是否有处理
- [ ] 权限不足是否有处理
- [ ] 数据不存在是否有处理
- [ ] 并发冲突是否有处理
- [ ] 第三方服务失败是否有处理

### 边界条件
- [ ] 输入字段的最大/最小值是否定义
- [ ] 空值/null 处理是否明确
- [ ] 特殊字符（emoji、HTML、SQL注入）处理是否说明
- [ ] 列表的空数据/海量数据场景是否考虑
- [ ] 时间相关的边界（跨天、跨时区、闰年）

### 状态覆盖
- [ ] 状态转换图中每条路径是否可测
- [ ] 是否存在"死状态"（进去了出不来）
- [ ] 异常中断后的状态恢复机制

---

## 四、UI/UX 设计师检查清单

### 交互一致性
- [ ] 操作模式是否与全局规范一致（弹窗、抽屉、新页面）
- [ ] 按钮样式和位置是否符合设计规范
- [ ] 表单校验的触发时机和提示方式是否统一
- [ ] 列表的排序、筛选、分页交互是否统一

### 状态完整性
- [ ] 空状态是否有设计（插画 + 文案 + 引导操作）
- [ ] 加载状态是否有设计（骨架屏 / loading / 占位符）
- [ ] 错误状态是否有设计（提示 + 重试/返回）
- [ ] 成功状态是否有反馈（Toast / 页面跳转）
- [ ] 禁用状态是否有视觉区分

### 信息架构
- [ ] 页面信息层级是否清晰（主操作突出、次操作收敛）
- [ ] 操作路径是否最短（核心任务几步完成）
- [ ] 用户是否能明确知道"我在哪、做了什么、下一步是什么"

### 无障碍
- [ ] 颜色对比度是否达标
- [ ] 是否仅依赖颜色传达信息（色盲友好）
- [ ] 可点击区域是否足够大（移动端 ≥ 44px）

### 设计周期
- [ ] 交互稿工作量
- [ ] 视觉稿工作量
- [ ] 标注和切图工作量

---

## 五、运营检查清单

### 数据与埋点
- [ ] 关键页面是否有 PV/UV 埋点
- [ ] 核心操作是否有事件埋点（点击、提交、完成）
- [ ] 转化漏斗的每一步是否可追踪
- [ ] 异常事件是否有埋点（便于监控）
- [ ] 埋点参数是否定义（事件名、属性、触发时机）

### 上线策略
- [ ] 是否有灰度发布计划（比例、时间、范围）
- [ ] 回滚方案是否明确
- [ ] 数据迁移方案是否考虑（如有）
- [ ] 新老版本兼容性是否处理

### 运营配置
- [ ] 文案是否可配置（无需发版修改）
- [ ] 功能开关是否需要（A/B 测试、紧急下线）
- [ ] 运营后台是否需要（内容管理、用户管理）
- [ ] 推送/通知策略是否合理

---

## 六、法务/合规检查清单

### 个人信息保护
- [ ] 收集了哪些个人信息？是否必要？
- [ ] 是否明确告知用户并获取同意
- [ ] 隐私政策是否需要更新
- [ ] 信息使用目的是否与收集目的一致

### 数据安全
- [ ] 敏感数据是否加密存储（密码、身份证、银行卡）
- [ ] 数据传输是否加密（HTTPS）
- [ ] 数据访问权限是否控制（最小权限原则）
- [ ] 数据留存期限是否合规
- [ ] 是否涉及跨境数据传输

### 内容合规
- [ ] UGC 内容是否有审核机制（先审后发 / 先发后审）
- [ ] 是否涉及第三方版权内容
- [ ] 是否有举报和处置机制
- [ ] 是否需要内容安全标注（年龄限制等）

### 用户协议
- [ ] 现有用户协议是否覆盖新功能
- [ ] 是否需要新增或修改条款
- [ ] 重大变更是否需要重新获取用户同意
